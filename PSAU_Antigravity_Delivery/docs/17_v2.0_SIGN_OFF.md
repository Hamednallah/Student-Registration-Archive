# v2.0 Sign-Off — Formal Acceptance Criteria
## The Gate Between Local Delivery and AWS Deployment

---

> **This document is a contract.**  
> Every checkbox must be verified and signed off before a single AWS resource is provisioned.  
> "I think it works" is not sign-off. "I ran this command and got this output" is sign-off.  
> 
> **v2.1 (AWS) does not start until every item below is checked.**

---

## How to Use This Document

For each checkbox:
1. Run the verification command
2. Paste the output (or describe what you saw) in the `Evidence` field
3. Check the box
4. A second engineer reviews the evidence

## After v2.0 Sign-Off: What Changes for v2.1

The code itself does not change between v2.0 and v2.1. Only the infrastructure changes:

| Component | v2.0 (Local) | v2.1 (AWS) |
|-----------|-------------|-----------|
| API hosting | Docker on local server | ECS Fargate |
| Database | PostgreSQL in Docker | RDS PostgreSQL Multi-AZ |
| Cache | Redis in Docker | ElastiCache Redis |
| Web hosting | nginx in Docker | S3 + CloudFront |
| Secrets | .env file | AWS Secrets Manager |
| Backups | Local disk + optional GDrive | S3 |
| Monitoring | Docker logs | CloudWatch |
| Domain | localhost / local IP | psau.edu.sd |
| TLS | Self-signed or none | ACM auto-renewed |
| Scaling | Manual Docker | ECS Auto Scaling |
