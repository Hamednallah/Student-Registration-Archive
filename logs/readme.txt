التقرير الفني الشامل لنظام أرشفة إيصالات تسجيل طلاب جامعة بورتسودان
مقدمة
يهدف هذا النظام إلى إدارة وأرشفة إيصالات تسجيل طلاب جامعة بورتسودان الوطنية. يتيح النظام للمستخدمين تسجيل وإدارة معلومات الطلاب والأقسام والإيصالات مع واجهة مستخدم سهلة الاستخدام وقاعدة بيانات آمنة.

نظام تسجيل الأحداث (Logging System)
### هيكل السجلات

1. **combined.log**: يحتوي على جميع الأحداث (INFO, WARN, ERROR)
2. **error.log**: يحتوي على الأخطاء فقط (ERROR)
3. **db-setup-*.log**: سجلات إعداد قاعدة البيانات

### مستويات التسجيل (Log Levels)

1. **INFO**: معلومات عامة عن تشغيل النظام
2. **WARN**: تحذيرات عن مشاكل محتملة
3. **ERROR**: أخطاء حرجة تؤثر على عمل النظام

### تناوب السجلات (Log Rotation)

- يتم إنشاء ملف سجل جديد يومياً
- يتم الاحتفاظ بالسجلات لمدة 30 يوماً
- يتم ضغط السجلات القديمة تلقائياً

### تنسيق السجلات

كل سجل يحتوي على:
- التاريخ والوقت
- مستوى السجل (INFO, WARN, ERROR)
- اسم الملف ورقم السطر
- الرسالة التفصيلية

### مثال على سجل

[2025-03-27T10:22:47+02:00] INFO: Server started on port 5000 (app.js:45)
[2025-03-27T10:23:12+02:00] ERROR: Database connection failed (database.js:78)

### إدارة السجلات

- يتم تخزين السجلات في مجلد /logs
- يمكن تكوين حجم ومدة الاحتفاظ بالسجلات عبر ملف .env
- يدعم النظام تصفية السجلات حسب النوع والتاريخ

### أفضل الممارسات

1. تسجيل جميع الأحداث المهمة
2. استخدام المستويات المناسبة لكل سجل
3. تضمين معلومات كافية للتحليل
4. حماية السجلات من الوصول غير المصرح به
5. مراجعة السجلات بانتظام لاكتشاف المشاكل

### تكوين السجلات

يمكن تكوين النظام عبر متغيرات البيئة:

- LOG_LEVEL: مستوى التسجيل (info, warn, error)
- LOG_ROTATION: تناوب السجلات (daily, weekly, monthly)
- LOG_RETENTION: مدة الاحتفاظ بالسجلات (بالأيام)

### تحليل السجلات

يمكن استخدام أدوات مثل:

- ELK Stack (Elasticsearch, Logstash, Kibana)
- Splunk
- Grafana Loki

### أمان السجلات

- تشفير السجلات الحساسة
- التحكم في الوصول إلى السجلات
- مراقبة محاولات الوصول غير المصرح به

### صيانة السجلات

- تنظيف السجلات القديمة تلقائياً
- مراقبة حجم السجلات
- نسخ السجلات احتياطياً بانتظام

آلية عمل النظام
1. تسجيل الدخول والمصادقة
CopyInsert
المستخدم --> إدخال بيانات الدخول --> التحقق من صحة البيانات --> مقارنة كلمة المرور المشفرة --> إنشاء رمز الجلسة --> الوصول إلى النظام
شرح: يقوم المستخدم بإدخال اسم المستخدم وكلمة المرور. يتحقق النظام من صحة البيانات ويقارن كلمة المرور المدخلة مع النسخة المشفرة في قاعدة البيانات. إذا كانت صحيحة، يتم إنشاء رمز جلسة للمستخدم والسماح له بالوصول إلى النظام.

2. إدارة الطلاب
CopyInsert
إضافة طالب --> التحقق من البيانات --> حفظ في قاعدة البيانات --> عرض قائمة الطلاب
شرح: يقوم المستخدم بإدخال بيانات الطالب (الاسم، الرقم، القسم، المستوى). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم عرض قائمة الطلاب والبحث عنهم وتعديل بياناتهم.

3. إدارة الإيصالات
CopyInsert
اختيار طالب --> إدخال بيانات الإيصال --> التحقق من البيانات --> حفظ في قاعدة البيانات --> طباعة الإيصال
شرح: يقوم المستخدم باختيار طالب من القائمة ثم إدخال بيانات الإيصال (رقم الإيصال البنكي، المبلغ، البنود المدفوعة). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم طباعة الإيصال أو استعراضه لاحقاً.

4. إدارة الأقسام
CopyInsert
إضافة قسم --> التحقق من البيانات --> حفظ في قاعدة البيانات --> عرض قائمة الأقسام
شرح: يقوم المستخدم بإدخال بيانات القسم (الاسم، عدد المستويات). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم عرض قائمة الأقسام وتعديلها.

5. النسخ الاحتياطي واستعادة البيانات
CopyInsert
طلب نسخة احتياطية --> إنشاء ملف النسخة --> حفظ الملف --> استعادة البيانات عند الحاجة
شرح: يمكن للمسؤول طلب إنشاء نسخة احتياطية من البيانات. يقوم النظام بإنشاء ملف يحتوي على جميع البيانات الحالية. يمكن استخدام هذا الملف لاستعادة البيانات في حالة فقدانها.

منطق التنفيذ والقرارات التصميمية
1. استخدام قاعدة بيانات Oracle
السبب: تم اختيار Oracle لقوتها وموثوقيتها في التعامل مع البيانات المؤسسية. توفر Oracle دعماً قوياً للمعاملات والاستعلامات المعقدة وأمان البيانات.

البديل المحتمل: كان يمكن استخدام MySQL أو PostgreSQL كبدائل أقل تكلفة، لكن تم تفضيل Oracle لتوافقها مع أنظمة الجامعة الأخرى.

2. استخدام Node.js/Express
السبب: يوفر Node.js أداءً عالياً للعمليات غير المتزامنة مثل قراءة وكتابة قاعدة البيانات. Express هو إطار عمل بسيط ومرن يسهل بناء واجهات برمجة التطبيقات.

البديل المحتمل: كان يمكن استخدام Java Spring أو .NET Core للحصول على أداء أفضل في بعض الحالات، لكن تم تفضيل Node.js لسرعة التطوير وسهولة الصيانة.

3. هيكل المشروع
السبب: تم اعتماد نمط MVC (النموذج-العرض-المتحكم) لفصل المكونات المختلفة للتطبيق. هذا يجعل الكود أكثر تنظيماً وقابلية للصيانة والاختبار.

مثال:

النماذج (Models): تتعامل مع قاعدة البيانات وتمثل هيكل البيانات
وحدات التحكم (Controllers): تحتوي على منطق الأعمال
المسارات (Routes): تحدد نقاط النهاية API
4. أمان النظام
الميزات المنفذة:

تشفير كلمات المرور باستخدام bcrypt
حماية من هجمات القوة الغاشمة باستخدام تحديد معدل الطلبات
تعزيز أمان HTTP باستخدام helmet
التحقق من صحة البيانات المدخلة
الخلاصة
نظام أرشفة إيصالات تسجيل طلاب جامعة بورتسودان هو نظام شامل يوفر حلاً متكاملاً لإدارة وأرشفة إيصالات تسجيل الطلاب. يتميز النظام بواجهة مستخدم سهلة الاستخدام وخادم قوي وقاعدة بيانات آمنة.

تم تصميم النظام باستخدام تقنيات حديثة مثل Node.js وExpress وOracle DB لضمان الأداء والأمان والموثوقية. يتبع النظام نمط MVC لتنظيم الكود وتسهيل الصيانة والتطوير المستقبلي.

يمكن تطوير النظام مستقبلاً بإضافة ميزات جديدة مثل:

تكامل مع نظام الدفع الإلكتروني
تطبيق للهاتف المحمول للطلاب
تحسين نظام التقارير والإحصائيات
أتمتة عملية النسخ الاحتياطي
هذا التقرير يوفر فهماً شاملاً لهيكل النظام ومكوناته وآلية عمله، مما يسهل على المطورين فهم النظام وصيانته وتطويره.

# التقرير الفني الشامل لنظام أرشفة إيصالات تسجيل طلاب جامعة بورتسودان

## مقدمة

يهدف هذا النظام إلى إدارة وأرشفة إيصالات تسجيل طلاب جامعة بورتسودان الوطنية. يتيح النظام للمستخدمين تسجيل وإدارة معلومات الطلاب والأقسام والإيصالات مع واجهة مستخدم سهلة الاستخدام وقاعدة بيانات آمنة.

## مخطط النظام

### المخطط العام للنظام (High-level Architecture)

```
+------------------+        +------------------+        +------------------+
|                  |        |                  |        |                  |
|  واجهة المستخدم   | <----> |  خادم التطبيق    | <----> |  قاعدة البيانات   |
|  (Frontend)      |        |  (Backend)       |        |  (Database)      |
|                  |        |                  |        |                  |
+------------------+        +------------------+        +------------------+
```

**المكونات الرئيسية:**
- **واجهة المستخدم**: تطبيق ويب مبني باستخدام HTML/CSS/JavaScript
- **خادم التطبيق**: خادم Node.js/Express يوفر واجهة برمجة التطبيقات (API)
- **قاعدة البيانات**: قاعدة بيانات Oracle مع مخطط SRRA

### المخطط التفصيلي للنظام (Low-level Architecture)

```
+------------------+        +------------------+        +------------------+
|   واجهة المستخدم  |        |    خادم التطبيق   |        |   قاعدة البيانات  |
+------------------+        +------------------+        +------------------+
| - صفحة الإيصالات  |        | - المصادقة       |        | - جدول الطلاب    |
| - صفحة الطلاب    | <----> | - التحقق        | <----> | - جدول الأقسام   |
| - صفحة الأقسام   |        | - المسارات      |        | - جدول الإيصالات |
| - صفحة التقارير  |        | - النماذج       |        | - جدول المستخدمين|
| - صفحة المستخدمين|        | - وحدات التحكم   |        | - جدول النسخ    |
+------------------+        +------------------+        |  الاحتياطية     |
                                                        +------------------+
```

**تدفق البيانات:**
1. يقوم المستخدم بتسجيل الدخول عبر واجهة المستخدم
2. يتم إرسال طلب إلى خادم التطبيق للمصادقة
3. يتحقق الخادم من بيانات المستخدم في قاعدة البيانات
4. بعد المصادقة، يمكن للمستخدم الوصول إلى وظائف النظام (إدارة الطلاب، الأقسام، الإيصالات)
5. يتم تخزين جميع البيانات في قاعدة البيانات Oracle

## هيكلة المشروع

### الدليل الرئيسي

```
/
├── server/           # خادم التطبيق (Node.js/Express)
│   ├── config/       # إعدادات التطبيق
│   ├── controllers/  # وحدات التحكم
│   ├── middleware/   # البرمجيات الوسيطة
│   ├── models/       # نماذج البيانات
│   ├── routes/       # مسارات API
│   ├── scripts/      # نصوص برمجية للإعداد والترحيل
│   ├── uploads/      # مجلد تحميل الملفات
│   ├── utils/        # أدوات مساعدة
│   ├── app.js        # نقطة دخول التطبيق
│   └── package.json  # تبعيات التطبيق
└── logs/             # سجلات النظام
```

### توثيق الملفات والمجلدات

#### 1. مجلد `server`

يحتوي على كود خادم التطبيق الذي يدير جميع عمليات النظام.

##### 1.1 ملف `app.js`

**الوصف**: نقطة الدخول الرئيسية للتطبيق. يقوم بإعداد خادم Express وتكوين البرمجيات الوسيطة وتوصيل قاعدة البيانات وتعريف المسارات.

**التبعيات الرئيسية**:
- `express`: إطار عمل لبناء تطبيقات الويب
- `body-parser`: لتحليل طلبات HTTP
- `helmet`: لتعزيز أمان التطبيق
- `cors`: للسماح بالاتصال من مصادر مختلفة
- `compression`: لضغط الاستجابات
- `dotenv`: لتحميل متغيرات البيئة

**الوظائف الرئيسية**:
- إعداد خادم Express
- تكوين البرمجيات الوسيطة للأمان
- إعداد CORS للسماح بالاتصال من مصادر محددة
- تكوين معالجة الطلبات والاستجابات
- توصيل قاعدة البيانات
- تعريف مسارات API
- إعداد معالجة الأخطاء
- إدارة إيقاف التشغيل الآمن للتطبيق

##### 1.2 مجلد `config`

**الوصف**: يحتوي على ملفات الإعدادات للتطبيق.

###### 1.2.1 ملف `database.js`

**الوصف**: يدير الاتصال بقاعدة البيانات Oracle.

**التبعيات**:
- `oracledb`: مكتبة للاتصال بقاعدة بيانات Oracle
- `dotenv`: لقراءة متغيرات البيئة

**الوظائف الرئيسية**:
- `initialize()`: تهيئة مجمع اتصالات قاعدة البيانات
- `close()`: إغلاق مجمع الاتصالات
- `getConnection()`: الحصول على اتصال من المجمع
- `executeQuery()`: تنفيذ استعلام SQL

##### 1.3 مجلد `models`

**الوصف**: يحتوي على نماذج البيانات التي تتفاعل مع قاعدة البيانات.

###### 1.3.1 ملف `index.js`

**الوصف**: نقطة دخول مركزية لجميع النماذج.

**النماذج المصدرة**:
- `UserModel`: نموذج المستخدمين
- `ReceiptModel`: نموذج الإيصالات
- `StudentModel`: نموذج الطلاب
- `DepartmentModel`: نموذج الأقسام

###### 1.3.2 ملف `userModel.js`

**الوصف**: يدير عمليات المستخدمين في قاعدة البيانات.

**التبعيات**:
- `bcrypt`: لتشفير كلمات المرور

**الوظائف الرئيسية**:
- `getAllUsers()`: استرجاع جميع المستخدمين
- `createUser()`: إنشاء مستخدم جديد
- `getUserById()`: استرجاع مستخدم بواسطة المعرف
- `getUserByUsername()`: استرجاع مستخدم بواسطة اسم المستخدم
- `updateUser()`: تحديث مستخدم
- `deleteUser()`: حذف مستخدم
- `verifyPassword()`: التحقق من كلمة المرور

##### 1.4 مجلد `routes`

**الوصف**: يحتوي على تعريفات مسارات API.

###### 1.4.1 ملف `index.js`

**الوصف**: يعرف جميع مسارات API للتطبيق.

**المسارات الرئيسية**:
- **مسارات المصادقة**:
  - `POST /api/login`: تسجيل الدخول
  - `POST /api/register`: تسجيل مستخدم جديد

- **مسارات الإيصالات**:
  - `GET /api/receipts`: استرجاع جميع الإيصالات
  - `POST /api/receipts`: إنشاء إيصال جديد
  - `GET /api/receipts/amounts`: استرجاع إجمالي المبالغ
  - `GET /api/receipts/recent`: استرجاع أحدث الإيصالات
  - `GET /api/receipts/:id`: استرجاع إيصال بواسطة المعرف
  - `GET /api/receipts/student/:studentId`: استرجاع إيصالات طالب
  - `GET /api/receipts/search`: البحث عن الإيصالات
  - `PUT /api/receipts/:id`: تحديث إيصال
  - `DELETE /api/receipts/:id`: حذف إيصال

- **مسارات الطلاب**:
  - `GET /api/students`: استرجاع جميع الطلاب
  - `POST /api/students`: إنشاء طالب جديد
  - `GET /api/students/stats`: استرجاع إحصائيات الطلاب
  - `GET /api/students/:id`: استرجاع طالب بواسطة المعرف
  - `PUT /api/students/:id`: تحديث طالب
  - `GET /api/students/department/:departmentId`: استرجاع الطلاب حسب القسم
  - `POST /api/students/search`: البحث عن الطلاب
  - `DELETE /api/students/:id`: حذف طالب

- **مسارات الأقسام**:
  - `GET /api/departments`: استرجاع جميع الأقسام
  - `POST /api/departments`: إنشاء قسم جديد
  - `GET /api/departments/:id`: استرجاع قسم بواسطة المعرف
  - `PUT /api/departments/:id`: تحديث قسم
  - `DELETE /api/departments/:id`: حذف قسم

- **مسارات المستخدمين (للمسؤولين فقط)**:
  - `GET /api/users`: استرجاع جميع المستخدمين
  - `POST /api/users`: إنشاء مستخدم جديد
  - `PUT /api/users/:id`: تحديث مستخدم
  - `DELETE /api/users/:id`: حذف مستخدم

- **مسارات النسخ الاحتياطي**:
  - `POST /api/backup`: إنشاء نسخة احتياطية
  - `POST /api/restore`: استعادة من نسخة احتياطية
  - `GET /api/backup/status`: استرجاع حالة النسخ الاحتياطي
  - `POST /api/backup/config`: تحديث إعدادات النسخ الاحتياطي

**الوصف**: يحتوي على ملفات سجلات النظام.

## التبعيات والخيارات التقنية

### الخادم (Backend)

| التبعية | الغرض | سبب الاختيار |
|---------|-------|-------------|
| Node.js | بيئة تشغيل JavaScript | سرعة التطوير، أداء عالي للعمليات غير المتزامنة |
| Express | إطار عمل الويب | بسيط، مرن، واسع الانتشار لبناء واجهات برمجة التطبيقات |
| Oracle DB | قاعدة البيانات | قوة، موثوقية، مناسبة للبيانات المؤسسية |
| bcrypt | تشفير كلمات المرور | أمان عالي لتخزين كلمات المرور |
| dotenv | إدارة متغيرات البيئة | فصل الإعدادات عن الكود لتحسين الأمان |
| helmet | تعزيز أمان HTTP | حماية التطبيق من ثغرات الويب الشائعة |
| cors | إدارة مشاركة الموارد عبر المنشأ | السماح بالاتصال الآمن بين الخادم وواجهة المستخدم |
| compression | ضغط الاستجابات | تحسين أداء التطبيق وتقليل استخدام النطاق الترددي |

## آلية عمل النظام

### 1. تسجيل الدخول والمصادقة

```
المستخدم --> إدخال بيانات الدخول --> التحقق من صحة البيانات --> مقارنة كلمة المرور المشفرة --> إنشاء رمز الجلسة --> الوصول إلى النظام
```

**شرح**: يقوم المستخدم بإدخال اسم المستخدم وكلمة المرور. يتحقق النظام من صحة البيانات ويقارن كلمة المرور المدخلة مع النسخة المشفرة في قاعدة البيانات. إذا كانت صحيحة، يتم إنشاء رمز جلسة للمستخدم والسماح له بالوصث إلى النظام.

### 2. إدارة الطلاب

```
إضافة طالب --> التحقق من البيانات --> حفظ في قاعدة البيانات --> عرض قائمة الطلاب
```

**شرح**: يقوم المستخدم بإدخال بيانات الطالب (الاسم، الرقم، القسم، المستوى). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم عرض قائمة الطلاب والبحث عنهم وتعديل بياناتهم.

### 3. إدارة الإيصالات

```
اختيار طالب --> إدخال بيانات الإيصال --> التحقق من البيانات --> حفظ في قاعدة البيانات --> طباعة الإيصال
```

**شرح**: يقوم المستخدم باختيار طالب من القائمة ثم إدخال بيانات الإيصال (رقم الإيصال البنكي، المبلغ، البنود المدفوعة). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم طباعة الإيصال أو استعراضه لاحقاً.

### 4. إدارة الأقسام

```
إضافة قسم --> التحقق من البيانات --> حفظ في قاعدة البيانات --> عرض قائمة الأقسام
```

**شرح**: يقوم المستخدم بإدخال بيانات القسم (الاسم، عدد المستويات). يتحقق النظام من صحة البيانات ويحفظها في قاعدة البيانات. يمكن للمستخدم عرض قائمة الأقسام وتعديلها.

### 5. النسخ الاحتياطي واستعادة البيانات

```
طلب نسخة احتياطية --> إنشاء ملف النسخة --> حفظ الملف --> استعادة البيانات عند الحاجة
```

**شرح**: يمكن للمسؤول طلب إنشاء نسخة احتياطية من البيانات. يقوم النظام بإنشاء ملف يحتوي على جميع البيانات الحالية. يمكن استخدام هذا الملف لاستعادة البيانات في حالة فقدانها.

## منطق التنفيذ والقرارات التصميمية

### 1. استخدام قاعدة بيانات Oracle

**السبب**: تم اختيار Oracle لقوتها وموثوقيتها في التعامل مع البيانات المؤسسية. توفر Oracle دعماً قوياً للمعاملات والاستعلامات المعقدة وأمان البيانات.

**البديل المحتمل**: كان يمكن استخدام MySQL أو PostgreSQL كبدائل أقل تكلفة، لكن تم تفضيل Oracle لتوافقها مع أنظمة الجامعة الأخرى.

### 2. استخدام Node.js/Express

**السبب**: يوفر Node.js أداءً عالياً للعمليات غير المتزامنة مثل قراءة وكتابة قاعدة البيانات. Express هو إطار عمل بسيط ومرن يسهل بناء واجهات برمجة التطبيقات.

**البديل المحتمل**: كان يمكن استخدام Java Spring أو .NET Core للحصول على أداء أفضل في بعض الحالات، لكن تم تفضيل Node.js لسرعة التطوير وسهولة الصيانة.

### 3. هيكل المشروع

**السبب**: تم اعتماد نمط MVC (النموذج-العرض-المتحكم) لفصل المكونات المختلفة للتطبيق. هذا يجعل الكود أكثر تنظيماً وقابلية للصيانة والاختبار.

**مثال**: 
- النماذج (Models): تتعامل مع قاعدة البيانات وتمثل هيكل البيانات
- وحدات التحكم (Controllers): تحتوي على منطق الأعمال
- المسارات (Routes): تحدد نقاط النهاية API

### 4. أمان النظام

**الميزات المنفذة**:
- تشفير كلمات المرور باستخدام bcrypt
- حماية من هجمات القوة الغاشمة باستخدام تحديد معدل الطلبات
- تعزيز أمان HTTP باستخدام helmet
- التحقق من صحة البيانات المدخلة

تم تصميم النظام باستخدام تقنيات حديثة مثل Node.js وExpress وOracle DB لضمان الأداء والأمان والموثوقية. يتبع النظام نمط MVC لتنظيم الكود وتسهيل الصيانة والتطوير المستقبلي.

يمكن تطوير النظام مستقبلاً بإضافة ميزات جديدة مثل:

- تطبيق للهاتف المحمول للطلاب
